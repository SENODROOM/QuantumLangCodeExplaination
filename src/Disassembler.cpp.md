# Disassembler.cpp

## Role in Compiler Pipeline

`Disassembler.cpp` is a crucial part of the Quantum Language compiler that converts bytecode back into human-readable assembly-like instructions. This process, called disassembly, facilitates debugging, comprehension, and optimization of the code generated by the compiler. The disassembler operates on the output produced by the bytecode generator, translating each opcode and operand into a clear and understandable format.

## Key Design Decisions and WHY

1. **Opcode Mapping**: A static function `opName` maps each opcode to its corresponding string representation. This allows for easy identification and display of the operations being performed during execution.

2. **Formatted Output**: The disassembler uses formatted output to ensure that each instruction is displayed clearly and consistently. This includes padding and alignment to make the disassembly easier to read.

3. **Instruction Representation**: Instructions are represented using a combination of opcodes and operands. The disassembler processes these representations to produce a readable format, making it possible to understand the sequence of operations without needing to consult the bytecode.

4. **Error Handling**: While not explicitly shown in the provided snippet, the disassembler should include robust error handling to manage cases where the bytecode might be malformed or incomplete.

## Major Classes/Functions Overview

### Function: `opName`
- **Purpose**: Maps opcodes to their string representations.
- **Parameters**:
  - `Op op`: The opcode to map.
- **Returns**: A string representing the opcode.

### Class: `Disassembler`
- **Purpose**: Handles the conversion of bytecode into human-readable assembly instructions.
- **Methods**:
  - `void disassemble(const std::vector<uint8_t> &bytecode)`: Main method that takes bytecode as input and outputs the disassembled instructions.
  - `std::string formatOperand(uint8_t operand)`: Formats the operand based on its type (e.g., immediate value, register index).

## Tradeoffs

1. **Readability vs. Performance**: The focus on readability makes the disassembler slower compared to a more optimized version that might sacrifice some clarity for performance gains.

2. **Memory Usage**: Storing the mapping between opcodes and strings requires additional memory, which could impact the overall memory footprint of the compiler.

3. **Complexity**: The need for formatted output adds complexity to the disassembler implementation, potentially increasing development time and maintenance costs.

Overall, `Disassembler.cpp` plays a vital role in the Quantum Language compiler by providing a clear and readable view of the bytecode, aiding developers in debugging and optimizing their programs. The design decisions made in this file prioritize readability and ease of use over performance and memory efficiency, reflecting a balance between functionality and usability in the context of software compilation.
