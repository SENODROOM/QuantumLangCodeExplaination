# Disassembler.cpp

## Role in Compiler Pipeline

`Disassembler.cpp` plays a vital role in the Quantum Language compiler's pipeline by converting bytecode back into human-readable assembly-like instructions. This process, known as disassembly, aids in debugging, comprehension, and optimization of the code generated by the compiler. The disassembler operates on the output produced by the bytecode generator and outputs it to an `std::ostream`.

## Key Design Decisions and Why

1. **OpCode Mapping**: A static function `opName` maps each opcode (`Op`) to its corresponding string representation. This mapping is essential for generating readable assembly instructions.

2. **Human-Readable Output**: By converting bytecode into assembly-like instructions, `Disassembler.cpp` ensures that the output is easily understandable by developers. This readability is particularly useful during debugging phases.

3. **Flexibility with Streams**: The use of `std::ostream` allows the disassembled output to be directed anywhere, such as standard output or a file. This flexibility makes it easy to integrate the disassembler into different parts of the compiler or external tools.

4. **Efficiency**: The disassembler processes bytecode efficiently, ensuring quick conversion without significant overhead. This is crucial for maintaining performance during compilation and debugging.

## Major Classes/Functions Overview

### Class: Disassembler

- **Purpose**: The primary class responsible for performing the disassembly process.
- **Methods**:
  - `disassemble(const Bytecode &bytecode)`: Main method that takes bytecode and outputs the disassembled instructions to the specified stream.
  - `printInstruction(std::ostream &os, const Instruction &instruction)`: Helper method to print individual instructions in a formatted manner.

### Function: opName

- **Purpose**: Maps an opcode to its string representation.
- **Parameters**:
  - `op`: An enumeration value representing the opcode.
- **Return Value**: A string representing the name of the opcode.

## Tradeoffs

1. **Readability vs. Performance**: While the disassembler aims for high readability, there might be slight performance implications due to the additional string operations involved. However, these trade-offs are generally acceptable for the benefits gained during debugging and comprehension.

2. **Complexity**: Adding more opcodes requires updating the `opName` function, which can introduce complexity. However, this complexity is manageable and necessary for supporting a rich set of language features.

3. **Integration**: The use of streams provides flexibility but may require additional setup when integrating the disassembler into other systems or environments.

Overall, `Disassembler.cpp` is a well-designed component of the Quantum Language compiler, balancing functionality, readability, and efficiency to support effective debugging and code analysis.
