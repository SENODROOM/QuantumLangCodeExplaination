# Quantum Language Compiler - Main Entry Point

## Overview

`src/main.cpp` serves as the primary entry point for the Quantum Language compiler. This file orchestrates the entire compilation process, managing the initialization and interaction between various components including the lexer, parser, interpreter, type checker, and error handler. It also handles command-line arguments and directs the execution flow based on user inputs.

## Key Design Decisions

### Modular Architecture

**WHY:** Implementing a modular architecture allows for better separation of concerns, making the codebase more maintainable and scalable. Each component can be developed, tested, and debugged independently, which facilitates parallel development and reduces complexity.

### Command-Line Argument Handling

**WHY:** Providing a robust mechanism for handling command-line arguments enables users to customize the behavior of the compiler according to their needs. For example, users can specify input files, enable test mode, or display help information.

### Error Handling

**WHY:** Comprehensive error handling is crucial for identifying and reporting issues during the compilation process. By centralizing error management, the compiler can provide clear and actionable feedback to the user, improving debugging and usability.

## Major Classes and Functions

### Lexer

**Purpose:** Converts raw text into tokens, which are the basic units of the language's syntax.

**Behavior:** Reads the source code character by character, identifies keywords, identifiers, operators, literals, and other tokens, and stores them in a token stream for further processing by the parser.

### Parser

**Purpose:** Constructs an abstract syntax tree (AST) from the token stream generated by the lexer.

**Behavior:** Uses a recursive descent parsing algorithm to traverse the token stream, building the AST based on the grammar rules of the quantum language. Errors are reported using the error handler component.

### Interpreter

**Purpose:** Executes the AST produced by the parser, providing dynamic evaluation of the quantum program.

**Behavior:** Traverses the AST and evaluates expressions, statements, and control structures. It supports interactive mode through the REPL (Read-Eval-Print Loop).

### Type Checker

**Purpose:** Ensures that the quantum program adheres to type safety rules.

**Behavior:** Analyzes the AST to verify variable types, function signatures, and other type-related constraints. If any type errors are detected, they are reported using the error handler.

### Error Handler

**Purpose:** Manages the reporting of errors encountered during the compilation process.

**Behavior:** Provides mechanisms for displaying error messages, warnings, and other diagnostic information to the user. It ensures that the compiler can gracefully handle errors without crashing.

### Value

**Purpose:** Represents values in the quantum language, including integers, floating-point numbers, strings, and booleans.

**Behavior:** Stores and manipulates data values used in the quantum program. It supports operations like arithmetic, comparison, and string concatenation.

## Tradeoffs and Limitations

- **Performance:** While modularity improves maintainability, it may introduce some overhead due to increased function calls and interface usage.
- **Complexity:** Supporting multiple features and ensuring type safety adds complexity to the compiler, potentially increasing the risk of bugs.
- **Resource Usage:** The dynamic nature of the interpreter requires additional memory and computational resources compared to a static compiler.

## Usage

To compile a quantum program, run the following command:

```sh
./quantum_compiler path/to/source.q
```

To enter interactive mode (REPL), simply execute:

```sh
./quantum_compiler
```

In test mode, use the `--test` flag:

```sh
./quantum_compiler --test path/to/test.q
```

This will bypass standard input and output, returning an empty result immediately.
