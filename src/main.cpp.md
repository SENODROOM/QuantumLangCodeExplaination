# Quantum Language Compiler - Main Entry Point

## Overview

`src/main.cpp` serves as the primary entry point for the Quantum Language compiler. It manages the overall compilation process by coordinating the initialization of essential components such as the lexer, parser, interpreter, type checker, and error handler. Additionally, it handles command-line arguments and controls the execution flow based on user inputs.

## Key Design Decisions

### Modular Architecture

**WHY:** Implementing a modular architecture was crucial because it allowed for better separation of concerns, making the codebase more maintainable and scalable. Each component (lexer, parser, etc.) could be developed independently and tested without affecting the others. This approach also facilitated easier debugging and future enhancements.

### Use of Command-Line Arguments

**WHY:** Incorporating support for command-line arguments enabled users to interact with the compiler directly from the terminal. This feature is particularly useful for testing, batch processing, and automating the compilation workflow. By handling these arguments in `main.cpp`, the compiler can adapt its behavior based on user preferences or requirements.

### Integration with External Libraries

**WHY:** Leveraging external libraries like `<filesystem>` provided robust functionality for file operations, which is essential for reading source files and writing output files. Using these libraries ensures cross-platform compatibility and simplifies the implementation of file handling features.

## Major Classes and Functions

### Lexer

**Purpose:** Converts raw text into tokens, which are the smallest meaningful units of the language.

**Behavior:** Reads the input source code and breaks it down into individual tokens, including keywords, identifiers, literals, and punctuation marks.

### Parser

**Purpose:** Constructs an abstract syntax tree (AST) from the sequence of tokens produced by the lexer.

**Behavior:** Analyzes the tokens using a recursive descent parsing algorithm to ensure they conform to the grammar rules of the Quantum Language. If errors are detected, the parser reports them to the error handler.

### Interpreter

**Purpose:** Executes the AST generated by the parser, providing dynamic evaluation of the program's logic.

**Behavior:** Traverses the AST and evaluates each node according to its type and content. This includes executing statements, evaluating expressions, and managing runtime data structures.

### Type Checker

**Purpose:** Validates the types used in the program to ensure consistency and prevent type-related errors.

**Behavior:** Checks the types of variables, function parameters, and return values against the defined grammar and type rules. Reports any type mismatches or other issues to the error handler.

### Error Handler

**Purpose:** Manages and reports errors encountered during the compilation process.

**Behavior:** Captures and displays error messages to the user, indicating the nature and location of the error within the source code. Provides suggestions for resolving common issues.

### Value Class

**Purpose:** Represents the values used in the program, including integers, strings, booleans, and custom data types.

**Behavior:** Encapsulates value manipulation and storage, ensuring type safety and proper memory management.

## Tradeoffs and Limitations

- **Modular Architecture vs. Complexity:** While a modular design enhances maintainability, it may introduce additional complexity in terms of inter-component communication and integration points.
  
- **Command-Line Argument Handling:** Although flexible, extensive command-line argument support requires careful parsing and validation to avoid misuse or unintended behavior.

- **External Library Dependency:** Relying on external libraries like `<filesystem>` adds dependencies but ensures platform-independent functionality. However, it might increase build times and complicate dependency management.

## Conclusion

`src/main.cpp` plays a pivotal role in orchestrating the Quantum Language compiler's operation. Its modular architecture, command-line argument handling, and integration with external libraries make it a versatile and efficient entry point. Despite some tradeoffs, this design decision enables the compiler to handle various tasks effectively and provides a solid foundation for future development.
