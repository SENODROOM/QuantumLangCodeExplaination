# Quantum Language Compiler - Main Entry Point

## Overview

`src/main.cpp` is the central entry point for the Quantum Language compiler, responsible for orchestrating the entire compilation process. This includes interactions with the lexer, parser, compiler, virtual machine (VM), disassembler, type checker, and error handler. The file also manages command-line arguments and directs the execution flow based on different build modes.

### Key Design Decisions and Why

The main entry point of the Quantum Language compiler is designed to handle various aspects of the compilation process efficiently. Here are some key design decisions:

1. **Modular Architecture**: Each component of the compiler (lexer, parser, compiler, VM, etc.) is encapsulated in its own class or function, promoting modularity and easier maintenance.
   
2. **Command-Line Argument Handling**: The program parses command-line arguments to determine the mode of operation (compiler, interpreter, or standalone bundled executable). This allows flexibility in how the compiler can be used.

3. **Error Handling**: A robust error handling system is implemented to manage and report errors during the compilation process. Errors are categorized and handled appropriately to ensure a smooth user experience.

4. **Embedded Bytecode Support**: For certain build modes, the program supports embedded bytecode within the executable. This feature enhances performance by avoiding external file I/O operations.

5. **Cross-Platform Compatibility**: While the code is primarily written for Windows, it includes conditional compilation directives to support other platforms if necessary.

### Major Classes/Functions Overview

#### `main()`
- **Role**: The primary function that initializes the compiler and processes command-line arguments.
- **Key Responsibilities**:
  - Parses command-line arguments to determine the mode of operation.
  - Initializes the appropriate components (lexer, parser, compiler, VM, etc.) based on the mode.
  - Orchestrates the compilation process, including lexing, parsing, compiling, and running the generated bytecode.
  
#### `getExecutablePath()`
- **Role**: Retrieves the path of the currently executing executable.
- **Key Responsibilities**:
  - Uses the Windows API (`GetModuleFileNameA`) to fetch the executable path.
  - Returns the path as a string.

#### `loadEmbeddedBytecode(const std::string &exePath)`
- **Role**: Loads embedded bytecode from the specified executable file.
- **Key Responsibilities**:
  - Opens the executable file in binary mode and checks its size.
  - Validates the magic number at the end of the file to ensure it contains valid bytecode.
  - Reads the payload size and the actual payload data.
  - Deserializes the payload into a `Chunk` object using the `Serializer` class.
  - Returns the deserialized `Chunk` object or `nullptr` if loading fails.

#### `printBanner()`
- **Role**: Prints the banner text for the Quantum Language compiler.
- **Key Responsibilities**:
  - Outputs a colorful ASCII art banner to the console.
  - Enhances the visual appeal and branding of the compiler.

### Tradeoffs

1. **Performance vs. Flexibility**: Supporting embedded bytecode enhances performance but limits the flexibility of the compiler's deployment options.
   
2. **Code Size vs. Execution Speed**: Embedding bytecode increases the size of the executable, which might affect cold-start times. However, it reduces runtime overhead associated with file I/O operations.

3. **Complexity vs. Simplicity**: The modular architecture simplifies the development and maintenance of individual components but adds complexity to the overall setup and configuration.

4. **Portability vs. Native Features**: The cross-platform compatibility allows the compiler to run on multiple systems but may limit access to native platform-specific features.

By carefully balancing these tradeoffs, `src/main.cpp` ensures that the Quantum Language compiler remains efficient, flexible, and easy to maintain while supporting various deployment scenarios.
