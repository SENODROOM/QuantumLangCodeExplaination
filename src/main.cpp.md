# Quantum Language Compiler - Main Entry Point

## Overview

`src/main.cpp` serves as the central entry point for the Quantum Language compiler, orchestrating the entire compilation process. This includes managing interactions with the lexer, parser, compiler, virtual machine (VM), disassembler, type checker, and error handler. It also handles command-line arguments and directs the execution flow based on the build mode defined (compiler, interpreter, or standalone bundled executable).

## Key Design Decisions and Why

### Modular Architecture
The compiler is designed with a modular architecture to facilitate maintenance and scalability. Each component (lexer, parser, compiler, VM, etc.) is encapsulated in its own class, promoting separation of concerns and allowing independent development and testing.

### Command-Line Argument Handling
`main.cpp` parses command-line arguments to determine the desired operation (compile, run, or bundle). This flexibility ensures that the compiler can be used in various scenarios, from simple script execution to complex project builds.

### Error Handling
Integrated error handling mechanisms ensure that any issues encountered during the compilation process are reported clearly and accurately. This helps users diagnose problems more efficiently and improve code quality.

### Platform-Specific Code
Platform-specific code, such as loading embedded bytecode on Windows, is isolated within `main.cpp`. This approach keeps the core logic platform-independent while providing necessary functionality for specific platforms.

## Major Classes/Functions Overview

### Lexer
- **Role**: Converts raw source code into tokens.
- **Overview**: The Lexer class processes the input source code and breaks it down into meaningful tokens, which are then consumed by the Parser.

### Parser
- **Role**: Constructs an abstract syntax tree (AST) from the sequence of tokens produced by the Lexer.
- **Overview**: The Parser class takes the tokens generated by the Lexer and constructs a structured AST, representing the syntactic structure of the source code.

### Compiler
- **Role**: Translates the AST into bytecode that can be executed by the VM.
- **Overview**: The Compiler class performs semantic analysis, generates intermediate representations, and finally translates these representations into bytecode.

### Virtual Machine (VM)
- **Role**: Executes the compiled bytecode.
- **Overview**: The VM class interprets or executes the bytecode depending on the build mode. It manages the runtime environment, including memory allocation, function calls, and control flow.

### Disassembler
- **Role**: Converts bytecode back into human-readable assembly code.
- **Overview**: The Disassembler class allows developers to inspect and debug the compiled bytecode by converting it into assembly-like instructions.

### Type Checker
- **Role**: Ensures that the types used in the source code are consistent and valid.
- **Overview**: The Type Checker class verifies the type correctness of expressions, variables, and functions, preventing common programming errors.

### Error Handler
- **Role**: Manages and reports errors during the compilation process.
- **Overview**: The Error Handler class provides mechanisms to detect, report, and handle errors, ensuring that the user is informed about any issues encountered.

### Value
- **Role**: Represents values in the Quantum Language.
- **Overview**: The Value class encapsulates different data types and provides methods for manipulating and accessing these values.

### Serializer
- **Role**: Serializes and deserializes bytecode.
- **Overview**: The Serializer class handles the conversion of bytecode to a binary format and vice versa, enabling embedding of bytecode within executables.

## Tradeoffs

### Performance vs. Flexibility
By using a modular architecture, the compiler sacrifices some performance due to additional function calls and overhead. However, this flexibility allows for easier updates and extensions, making the compiler more robust over time.

### Platform Independence vs. Specific Features
While the core logic of the compiler remains platform-independent, certain features like embedded bytecode loading are implemented platform-specifically. This decision balances the need for cross-platform compatibility with the inclusion of useful platform-specific functionalities.

### Memory Usage vs. Execution Speed
The VM's interpretation of bytecode can lead to higher memory usage compared to direct bytecode execution. However, this trade-off ensures that the compiler remains accessible and usable across different environments without requiring specialized hardware support.

In summary, `src/main.cpp` plays a crucial role in the Quantum Language compiler by coordinating all components and handling essential tasks such as command-line argument parsing and error management. Its modular design and platform-specific optimizations contribute to both flexibility and performance, making it a versatile tool for developing and executing Quantum Language programs.
