# QuantumLanguage Compiler - Interpreter.h

## Overview

The `include/Interpreter.h` header file is a crucial component of the QuantumLanguage compiler, responsible for interpreting and executing the abstract syntax tree (AST) generated by the parser. This file defines the `Interpreter` class, which handles the evaluation of expressions, execution of statements, and management of function calls. It leverages C++'s type system and features such as `std::variant` to encapsulate various data types and ensure flexibility in handling different kinds of operations.

## Key Design Decisions

### Use of `std::variant` for Value Encapsulation

**WHY:** The choice to use `std::variant` over traditional unions allows for compile-time type safety and easier management of multiple data types. This eliminates runtime errors associated with union misuse and makes the code more robust and maintainable.

### Environment Management with `std::shared_ptr`

**WHY:** Using `std::shared_ptr` for environment management facilitates memory sharing and automatic garbage collection. This approach simplifies the process of creating nested scopes and ensures that environments are properly deallocated when they go out of scope, preventing memory leaks.

### Limiting Step Count to Prevent Infinite Loops

**WHY:** To guard against potential infinite loops, especially in cases where user input might not be controlled, the interpreter limits the number of steps it can execute. This prevents the program from running indefinitely and helps in maintaining stability during compilation and execution.

## Class and Function Documentation

### Interpreter Class

**Purpose:** Manages the interpretation and execution of the AST nodes.

**Behaviour:**
- **Constructor (`Interpreter()`):** Initializes the interpreter with default settings.
- **execute (`void execute(ASTNode &node)`):** Executes the given AST node.
- **evaluate (`QuantumValue evaluate(ASTNode &node)`):** Evaluates the given AST node and returns its result.
- **execBlock (`void execBlock(BlockStmt &s, std::shared_ptr<Environment> scope = nullptr)`):** Executes a block statement within a specified scope.
- **registerNatives (`void registerNatives()`):** Registers native functions and methods available in the interpreter.
- **Statement Executors:** Methods like `execVarDecl`, `execFunctionDecl`, etc., handle specific types of statements.
- **Expression Evaluators:** Methods like `evalBinary`, `evalUnary`, etc., evaluate specific types of expressions.
- **C++ Pointer Evaluators:** Methods like `evalAddressOf`, `evalDeref`, etc., handle pointer-related operations.
- **callFunction (`QuantumValue callFunction(std::shared_ptr<QuantumFunction> fn, std::vector<QuantumValue> args)`):** Calls a user-defined function.
- **callNative (`QuantumValue callNative(std::shared_ptr<QuantumNative> fn, std::vector<QuantumValue> args)`):** Calls a native function.
- **callInstanceMethod (`QuantumValue callInstanceMethod(std::shared_ptr<QuantumInstance> inst, std::shared_ptr<QuantumFunction> fn, std::vector<QuantumValue> args)`):** Calls a method on an instance object.
- **Built-in Method Dispatch:** Methods like `callMethod`, `callArrayMethod`, etc., dispatch built-in methods based on the type of the object.
- **setLValue (`void setLValue(ASTNode &target, QuantumValue val, const std::string &op)`):** Sets the value of an l-value expression using the provided operator.

### QuantumValue Class

**Purpose:** Represents a quantum value, capable of holding any of the supported data types.

**Trade-offs:** While `std::variant` provides flexibility, it may introduce performance overhead compared to simpler types. Additionally, error handling becomes more complex due to the dynamic nature of type checking.

### Environment Class

**Purpose:** Manages symbol tables and scopes for variables, functions, and classes.

**Trade-offs:** Memory management is simplified through `std::shared_ptr`, but it may lead to increased overhead if not used efficiently.

## Tradeoffs and Limitations

- **Performance Overhead:** The use of `std::variant` and `std::shared_ptr` introduces some performance overhead, which could become significant in large-scale applications.
- **Complexity:** Managing multiple data types and nested scopes increases the complexity of the codebase, making it harder to debug and maintain.
- **Memory Usage:** Automatic garbage collection via `std::shared_ptr` can lead to higher memory usage, particularly in scenarios with many nested scopes.
