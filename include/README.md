# QuantumLanguage Compiler - Interpreter.h

## Overview

The `include/Interpreter.h` header file is a crucial component of the QuantumLanguage compiler, responsible for interpreting and executing the Abstract Syntax Tree (AST) generated by the parser. This file defines the `Interpreter` class, which handles the evaluation of expressions, execution of statements, and management of the runtime environment. It forms the backbone of the compiler's interpretation phase, ensuring that the code written in QuantumLanguage is executed correctly and efficiently.

### Role in Compiler Pipeline

1. **Parsing**: The AST is constructed by parsing the source code.
2. **Interpretation**: The `Interpreter` class evaluates the AST nodes and executes the corresponding operations.
3. **Execution**: Statements like variable declarations, function calls, and control flow structures are handled by specific methods within the `Interpreter`.

### Key Design Decisions and Why

1. **Separation of Concerns**:
   - **Statement Executors**: Methods like `execVarDecl`, `execFunctionDecl`, etc., handle the execution of different statement types.
   - **Expression Evaluators**: Functions such as `evalBinary`, `evalUnary`, etc., manage the evaluation of expressions.
   - This separation allows for modular and maintainable code.

2. **Runtime Environment Management**:
   - The `env` member variable represents the current execution context or scope.
   - The `globals` member variable holds the global scope, enabling access to global variables throughout the program.

3. **Infinite Loop Protection**:
   - The `stepCount_` counter ensures that the interpreter does not run indefinitely by limiting the number of steps it can take.
   - This helps prevent potential crashes due to infinite loops, especially in edge cases or malformed input.

4. **Built-in Method Dispatch**:
   - Specific methods like `callArrayMethod`, `callStringMethod`, etc., handle built-in methods for different data types.
   - This approach centralizes method handling, making it easier to extend and modify.

### Major Classes/Functions Overview

#### Class: `Interpreter`

- **Constructor**: Initializes the interpreter with a default global scope.
- **Methods**:
  - `execute(ASTNode &node)`: Executes the entire AST starting from the given node.
  - `evaluate(ASTNode &node)`: Evaluates the value of the given AST node.
  - `execBlock(BlockStmt &s, std::shared_ptr<Environment> scope = nullptr)`: Executes a block of statements within a specified scope.
  - `registerNatives()`: Registers native functions and methods available to the interpreter.
  - `setLValue(ASTNode &target, QuantumValue val, const std::string &op)`: Sets the value of a left-hand side expression based on the operation performed.

#### Expression Evaluation Functions

- **Binary Expressions**: Evaluated using `evalBinary(BinaryExpr &e)`.
- **Unary Expressions**: Evaluated using `evalUnary(UnaryExpr &e)`.
- **Assignment Expressions**: Evaluated using `evalAssign(AssignExpr &e)`.
- **Function Calls**: Evaluated using `evalCall(CallExpr &e)`.
- **Indexing Expressions**: Evaluated using `evalIndex(IndexExpr &e)`.
- **Member Access Expressions**: Evaluated using `evalMember(MemberExpr &e)`.
- **Array Literals**: Evaluated using `evalArray(ArrayLiteral &e)`.
- **Dictionary Literals**: Evaluated using `evalDict(DictLiteral &e)`.
- **Lambda Expressions**: Evaluated using `evalLambda(LambdaExpr &e)`.
- **List Comprehensions**: Evaluated using `evalListComp(ListComp &e)`.
- **Identifiers**: Evaluated using `evalIdentifier(Identifier &e)`.

#### C++ Pointer Handling Functions

- **Address-of Operator**: Evaluated using `evalAddressOf(AddressOfExpr &e)`.
- **Dereference Operator**: Evaluated using `evalDeref(DerefExpr &e)`.
- **Arrow Operator**: Evaluated using `evalArrow(ArrowExpr &e)`.
- **New Expression**: Evaluated using `evalNewExpr(NewExpr &e)`.

#### Function Call Handlers

- `callFunction(std::shared_ptr<QuantumFunction> fn, std::vector<QuantumValue> args)`: Handles calling user-defined functions.
- `callNative(std::shared_ptr<QuantumNative> fn, std::vector<QuantumValue> args)`: Handles calling native functions.
- `callInstanceMethod(std::shared_ptr<QuantumInstance> inst, std::shared_ptr<QuantumFunction> fn, std::vector<QuantumValue> args)`: Handles calling instance methods.

### Tradeoffs

1. **Performance vs. Flexibility**:
   - The use of virtual functions for expression and statement evaluation provides flexibility but may introduce some performance overhead compared to direct method calls.

2. **Memory Usage**:
   - Managing dynamic memory through smart pointers (`std::shared_ptr`) helps reduce memory leaks and improves overall memory usage efficiency.

3. **Complexity vs. Simplicity**:
   - Separating concerns into distinct classes and functions increases complexity but makes the codebase more organized and easier to understand and maintain.

By carefully designing these components, the `Interpreter.h` file ensures that the QuantumLanguage compiler can interpret and execute complex programs efficiently while maintaining robustness and scalability.
