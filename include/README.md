# QuantumLanguage Compiler - TypeChecker.h

## Overview

The `include/TypeChecker.h` header file is an essential part of the QuantumLanguage compiler, focusing on the type checking mechanism. The type checker ensures that the code adheres to the declared types, preventing runtime errors due to type mismatches. It plays a crucial role in validating the syntactic correctness of expressions and statements during the compilation process.

## Role in Compiler Pipeline

The type checker operates as a critical stage within the compiler's pipeline. It follows the parsing phase but precedes the code generation phase. During this stage, it traverses the abstract syntax tree (AST) generated by the parser and checks the types of all expressions and variables. If any type inconsistencies are detected, it raises a `StaticTypeError`, which includes the error message and the line number where the inconsistency occurred. This allows developers to identify and fix type-related issues before proceeding with further stages of compilation.

## Key Design Decisions and Why

1. **Exception Handling**: The type checker uses exceptions (`StaticTypeError`) to handle type errors. This approach simplifies error reporting and recovery, ensuring that the compiler can halt processing immediately upon encountering a type mismatch.

2. **Type Environment**: A `TypeEnv` struct is used to manage the scope of variable types. Each `TypeEnv` instance contains a map of variable names to their types and a pointer to its parent environment. This hierarchical structure allows the type checker to handle nested scopes effectively, resolving variable types based on their nearest definition.

3. **Recursive Type Checking**: The `checkNode` method performs recursive type checking on each node in the AST. This ensures that every expression and statement is validated against its expected type, maintaining consistency throughout the entire program.

## Major Classes/Functions Overview

### Class: `StaticTypeError`
- Inherits from `std::runtime_error`.
- Constructor takes an error message and a line number.
- Used to report static type errors encountered during the type checking process.

### Struct: `TypeEnv`
- Manages the scope of variable types.
- Contains a map of variable names to their types and a shared pointer to the parent environment.
- Provides methods to define new variables and resolve existing ones.

### Class: `TypeChecker`
- Initializes the type checker with a global type environment.
- Offers a public method `check` to validate an entire list of AST nodes or a single node.
- Utilizes the `checkNode` method to recursively validate each node in the AST.

## Tradeoffs

1. **Performance vs. Accuracy**: While using exceptions for error handling enhances accuracy by allowing immediate detection of type errors, it may introduce performance overhead compared to other error-handling mechanisms. However, the benefits of early error detection outweigh the potential cost in most practical scenarios.

2. **Complexity vs. Maintainability**: Implementing a hierarchical type environment adds complexity to the type checker. However, this design improves maintainability by clearly separating different scopes and making it easier to extend the type system in the future.

3. **Memory Usage**: Storing type environments hierarchically can lead to increased memory usage, especially in large programs with many nested scopes. However, this tradeoff is generally acceptable given the importance of accurate type information for effective compilation.

By leveraging these features and addressing the associated tradeoffs, the `TypeChecker.h` header file provides a robust foundation for ensuring type safety in the QuantumLanguage compiler, ultimately contributing to the reliability and efficiency of the compiled code.
