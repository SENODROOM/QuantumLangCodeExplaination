# QuantumLanguage Compiler - Compiler.h

## Overview

The `include/Compiler.h` header file forms a crucial component of the QuantumLanguage compiler, focusing on the definition and implementation of the compiler itself. This compiler translates abstract syntax trees (ASTs) into executable chunks, managing various aspects such as scope resolution, function handling, and class definitions.

## Role in Compiler Pipeline

The compiler operates within the broader context of the QuantumLanguage compiler pipeline. It takes an AST generated by the parser and converts it into a series of bytecode instructions encapsulated in a `Chunk`. The `Chunk` represents a sequence of operations that can be executed by the virtual machine (VM). The compiler ensures that all variables, functions, and classes are properly defined and referenced throughout the program.

## Key Design Decisions and Why

### Scope Management

- **Why**: Efficiently managing variable scopes is critical for preventing conflicts and ensuring correct variable access.
- **Design Decision**: The compiler uses nested structures (`CompilerState`) to represent different scopes. Each `CompilerState` contains information about local variables, upvalues, and the current scope depth.

### Bytecode Emission

- **Why**: Generating bytecode allows for efficient execution by the VM, which interprets these instructions.
- **Design Decision**: The compiler provides helper functions like `emit`, `emitJump`, and `patchJump` to manage the emission and patching of bytecode instructions. These functions handle details such as operand calculation and instruction placement.

### Function Compilation

- **Why**: Functions need special handling to ensure proper stack management and control flow.
- **Design Decision**: The compiler includes specific methods (`compileFunctionDecl`) to handle function declarations, including parameter parsing and local variable initialization.

### Class Compilation

- **Why**: Classes require comprehensive support for inheritance, method definitions, and instance creation.
- **Design Decision**: The compiler features a dedicated method (`compileClassDecl`) to handle class declarations, managing fields, methods, and constructor logic.

## Major Classes/Functions Overview

### Compiler Class

- **Overview**: Manages the overall compilation process, converting AST nodes into bytecode.
- **Key Methods**:
  - `compile(ASTNode &root)`: Compiles an entire program starting from the root AST node.
  - `compileNode(ASTNode &node)`: Compiles individual AST nodes based on their type.
  - `compileBlock(BlockStmt &b)`: Handles block statements, typically used in control structures like loops and conditionals.
  - `compileExpr(ASTNode &node)`: Compiles expressions, evaluating them and generating corresponding bytecode.

### CompilerState Struct

- **Overview**: Represents the state of the compiler at any given point, including the current chunk, local variables, and upvalues.
- **Key Members**:
  - `chunk`: A shared pointer to the `Chunk` being compiled.
  - `locals`: A vector of `Local` structs representing local variables.
  - `upvalues`: A vector of `UpvalueDesc` structs representing upvalues.
  - `scopeDepth`: Tracks the current scope depth.
  - `enclosing`: Points to the enclosing `CompilerState`.

### Local Struct

- **Overview**: Describes a local variable, including its name, depth, and whether it is captured as an upvalue.
- **Key Members**:
  - `name`: The name of the local variable.
  - `depth`: The depth of the scope where the variable is declared.
  - `isCaptured`: Indicates if the variable has been promoted to an upvalue.

### UpvalueDesc Struct

- **Overview**: Describes an upvalue, indicating whether it captures a local variable from the enclosing scope and its index.
- **Key Members**:
  - `isLocal`: True if the upvalue captures a local variable from the directly enclosing scope.
  - `index`: The slot index of the local variable or upvalue index in the outer scope.

## Tradeoffs

### Memory Usage vs. Performance

- **Tradeoff**: Managing multiple `CompilerState` instances requires additional memory but improves performance by allowing for efficient scope resolution and bytecode generation.
- **Solution**: Using smart pointers (`std::shared_ptr`) minimizes memory overhead while maintaining strong ownership semantics.

### Code Complexity vs. Flexibility

- **Tradeoff**: Supporting complex features like classes and inheritance increases code complexity but enhances the language's flexibility and expressiveness.
- **Solution**: Carefully designed helper functions and methods simplify the main compilation logic, making the codebase more maintainable.

### Error Handling vs. Usability

- **Tradeoff**: Comprehensive error handling mechanisms improve robustness but may complicate user experience with verbose error messages.
- **Solution**: Balancing thorough error checking with clear, concise feedback through custom exception types and error reporting mechanisms.

In summary, the `include/Compiler.h` header file plays a pivotal role in the QuantumLanguage compiler by defining the core functionality required to translate ASTs into executable bytecode. Through strategic design decisions, it balances efficiency, flexibility, and usability, ensuring that the compiler can effectively handle a wide range of quantum programming constructs.
