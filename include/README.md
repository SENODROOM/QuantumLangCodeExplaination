# QuantumLanguage Compiler - Opcode.h

## Overview

The `include/Opcode.h` header file defines the set of operations (opcodes) used by the QuantumLanguage virtual machine (VM). These opcodes are fundamental to the execution process of the compiler-generated bytecode, enabling efficient and accurate program execution.

### Role in Compiler Pipeline

1. **Bytecode Generation**: The compiler translates high-level quantum language constructs into these opcodes, forming the executable bytecode.
2. **Execution Engine**: During runtime, the VM interprets and executes these opcodes, managing the program's state and control flow.

### Key Design Decisions and Why

- **Minimalistic Approach**: By keeping the opcode set minimal, the compiler can generate smaller bytecode files, improving performance and reducing memory usage.
- **Extensibility**: The inclusion of C++ extension opcodes (`ADDRESS_OF`, `DEREF`, `ARROW`) allows the VM to interact directly with native C++ data structures, enhancing flexibility and performance for certain use cases.
- **Exception Handling**: Comprehensive exception handling opcodes (`PUSH_HANDLER`, `POP_HANDLER`, `RAISE`, `RERAISE`) ensure robust error management within the program.
- **Dynamic Data Structures**: Operations like `MAKE_ARRAY`, `MAKE_DICT`, and `MAKE_TUPLE` support dynamic data structures, enabling flexible and powerful programming paradigms.

### Major Classes/Functions Overview

#### Enum Class `Op`
This enumeration class represents all possible opcodes used by the VM. Each opcode corresponds to a specific operation or action that the VM performs during execution.

#### Functions
- **Stack Manipulation**: `LOAD_CONST`, `LOAD_NIL`, `LOAD_TRUE`, `LOAD_FALSE`, `POP`
- **Variables**: `DEFINE_GLOBAL`, `LOAD_GLOBAL`, `STORE_GLOBAL`, `DEFINE_LOCAL`, `LOAD_LOCAL`, `STORE_LOCAL`, `DEFINE_CONST`
- **Upvalue/Closure Captures**: `LOAD_UPVALUE`, `STORE_UPVALUE`, `CLOSE_UPVALUE`
- **Arithmetic**: `ADD`, `SUB`, `MUL`, `DIV`, `MOD`, `FLOOR_DIV`, `POW`, `NEG`
- **Bitwise**: `BIT_AND`, `BIT_OR`, `BIT_XOR`, `BIT_NOT`, `LSHIFT`, `RSHIFT`
- **Comparison**: `EQ`, `NEQ`, `LT`, `LTE`, `GT`, `GTE`
- **Logical**: `NOT`, `AND`, `OR`
- **String**: `CONCAT`
- **Control Flow**: `JUMP`, `JUMP_IF_FALSE`, `JUMP_IF_TRUE`, `LOOP`, `JUMP_ABSOLUTE`
- **Function/Call**: `CALL`, `RETURN`, `RETURN_NIL`, `MAKE_FUNCTION`, `MAKE_CLOSURE`
- **Collections**: `MAKE_ARRAY`, `MAKE_DICT`, `MAKE_TUPLE`
- **Member/Index Access**: `GET_INDEX`, `SET_INDEX`, `GET_MEMBER`, `SET_MEMBER`, `GET_SUPER`
- **Iteration**: `FOR_ITER`, `MAKE_ITER`
- **Classes**: `MAKE_CLASS`, `INHERIT`, `BIND_METHOD`, `INSTANCE_NEW`
- **Exceptions**: `PUSH_HANDLER`, `POP_HANDLER`, `RAISE`, `RERAISE`
- **Pointers (C++ Extensions)**: `ADDRESS_OF`, `DEREF`, `ARROW`
- **Miscellaneous**: `PRINT`

### Tradeoffs

- **Performance vs. Flexibility**: While the minimalistic approach aims for better performance, including C++ extension opcodes increases flexibility but may introduce complexity.
- **Memory Usage vs. Execution Speed**: Smaller bytecode files lead to reduced memory usage, which can improve execution speed on systems with limited resources.
- **Simplicity vs. Robustness**: A simpler opcode set might be easier to implement, but a more comprehensive one ensures better robustness and functionality.

By understanding these opcodes and their roles, developers can gain insight into how the QuantumLanguage compiler generates and executes bytecode, facilitating debugging and optimization efforts.
